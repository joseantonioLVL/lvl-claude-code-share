Sección 1 — Fundamentos

¿Qué es Claude Code y por qué te importa?

La diferencia entre tener un asistente que conversa y uno que trabaja con tus archivos reales.

🎯

Lo que vas a aprender: En qué se diferencia Claude Code de claude.ai, cuándo usar cada uno, y por qué Claude Code es la herramienta que transforma el análisis cualitativo del equipo.

Cuando abres claude.ai en el navegador, estás chateando con una IA. Le pegas texto, le haces preguntas, te responde. Útil. Pero limitado: cada conversación empieza desde cero, no puede leer tus archivos locales, y si tu proyecto tiene 30 transcripciones, tendrías que pegarlas una por una.

Claude Code es diferente. Es una herramienta que corre en tu computadora y tiene acceso directo a todos tus archivos. Puede leer 30 transcripciones de una vez, escribir análisis en carpetas específicas, recordar el contexto de tu proyecto a través del CLAUDE.md, y ejecutar flujos de trabajo completos con un solo comando.

Piénsalo así: Claude web es como un consultor que atiendes por videollamada. Claude Code es como un analista senior que se sienta al lado tuyo y trabaja directamente con tu computadora.

🌐 Usa Claude Web cuando...

  • Tienes una pregunta rápida
  • Quieres hacer brainstorming de ideas
  • Necesitas redactar un email o documento corto
  • Estás explorando un tema nuevo
  • No tienes acceso a la terminal

💻 Usa Claude Code cuando...

  • Tienes múltiples transcripciones que analizar
  • Quieres resultados consistentes y reproducibles
  • Necesitas que el análisis vaya a archivos específicos
  • Trabajas en equipo y quieren compartir el método
  • Quieres automatizar tareas repetitivas
💡
Se complementan, no compiten

El flujo ideal: usa Claude Web para estructurar tus preguntas de investigación, luego Claude Code para analizar las transcripciones, y de vuelta a Claude Web para pulir los entregables. Son herramientas distintas para momentos distintos.

¿Por qué esto importa para Victoria Lab?

En un proyecto típico de investigación cualitativa, el cuello de botella no es tener las transcripciones — es procesarlas con rigor y consistencia. Sin Claude Code, el análisis depende de quién lo hace, de cuánto tiempo tiene, y de cuánto puede leer en una sentada.

Con Claude Code, el proyecto tiene memoria (CLAUDE.md), reglas (.claude/rules/), y flujos de trabajo repetibles (skills). Cuando onboardeas a alguien nuevo, no tienes que explicarle tu metodología — está en los archivos.

Sección 2 — Instalación

Setup en 10 minutos

Tutorial paso a paso para gente que nunca ha usado la terminal. Si puedes copiar y pegar, puedes hacer esto.

🎯

Lo que vas a aprender: Cómo abrir la terminal en Mac, instalar Claude Code, y lanzar tu primera sesión de análisis. No necesitas saber programar.

⚠️
Antes de empezar

Necesitas: Mac con macOS 12 o superior · Conexión a internet · Una cuenta en claude.ai (gratis o de pago)

  • Abre la Terminal

    La Terminal es una ventana donde escribes comandos de texto. No es tan complicada como parece.

    💡

    Presiona ⌘ + Espacio para abrir Spotlight, escribe "Terminal" y presiona Enter. También la encuentras en Aplicaciones → Utilidades → Terminal.

    Verás una ventana negra o blanca con un cursor parpadeante. Eso es la terminal. Está esperando que escribas algo.

  • Instala Claude Code

    Copia y pega este comando en la terminal, exactamente como está, y presiona Enter:

    Terminal
    $ curl -fsSL https://claude.ai/install.sh | bash

✓ Claude Code instalado correctamente

    El proceso toma 10-30 segundos. Verás texto desfilando — es normal. Cuando vuelva a aparecer el cursor ($), terminó.

  • Navega a tu carpeta de proyecto

    Antes de iniciar Claude, tienes que estar en la carpeta correcta. Por ejemplo, si tu proyecto está en Documentos:

    Terminal
    $ cd ~/Documents/mi-proyecto-whoop

# cd = "change directory" = cambiar de carpeta
# ~/Documents = la carpeta Documentos de tu Mac
# Tip: puedes arrastrar la carpeta desde Finder
#      directamente a la terminal después de escribir "cd "
    💡

    Atajo: Abre Finder, navega a tu carpeta de proyecto, y arrastra la carpeta directamente a la ventana de la terminal después de escribir cd (con espacio). Se auto-completa la ruta.

  • Lanza Claude Code

    Escribe claude y presiona Enter:

    Terminal
    $ claude

Comandos básicos que necesitas saber

🔑 Atajos de teclado en Claude Code

  • Enter — envía tu mensaje
  • Shift+Enter — nueva línea sin enviar
  • ↑ flecha arriba — historial de comandos
  • Ctrl+C — cancela lo que está haciendo
  • /help — muestra la ayuda
  • /compact — comprime el contexto (útil en sesiones largas)
  • /exit — cierra Claude Code

📂 Comandos de terminal que usarás

  • cd [carpeta] — navegar a una carpeta
  • ls — listar archivos en la carpeta actual
  • pwd — ver en qué carpeta estás
  • mkdir [nombre] — crear carpeta
  • claude — iniciar Claude Code
  • claude --resume — retomar sesión anterior

Sección 3 — Análisis con IA

Los 5 errores del análisis con IA

Por qué el análisis con IA puede ser peor que útil — y cómo evitarlo con evidencia real del proyecto Whoop.

🎯

Lo que vas a aprender: Los 5 patrones de falla más comunes cuando se usa IA para análisis cualitativo, con ejemplos concretos de cómo se ven — y cómo los detectas antes de que lleguen al cliente.

🏷️

Nota sobre los ejemplos: En esta sección usamos datos del proyecto Whoop Screen Exploration Q4 2025 (un estudio de investigación de usuarios de wearables fitness) como ejercicio de entrenamiento. Los mismos principios aplican exactamente a tus proyectos de LVL — conversaciones de bot de WhatsApp, encuestas de need assessment, o cualquier investigación cualitativa con clientes.

Usar IA para analizar investigación cualitativa es poderoso — pero solo si sabes qué puede salir mal. Estos cinco errores aparecen repetidamente en proyectos reales. Cada uno tiene una causa específica y una solución concreta.

1

Evidencia inventada (alucinación)

La IA cita datos o frases que no existen en las transcripciones

›
❌
Qué es

La IA genera una "cita" que suena plausible pero no aparece en ninguno de tus archivos. O afirma que "el 70% de participantes mencionó X" cuando en realidad fue uno solo.

⚠️
Por qué pasa

Los modelos de lenguaje completan patrones. Si le preguntas "¿qué dijo P03 sobre las notificaciones?", el modelo puede generar una respuesta coherente aunque P03 nunca mencionó notificaciones. La IA prefiere responder que admitir que no tiene evidencia.

Ejemplo real: al analizar las transcripciones Whoop, un análisis sin anclaje en datos podría generar:

❌ Output con alucinación:

"P03 mencionó explícitamente que las notificaciones push fueron la razón principal de su cancelación." — Esta cita no existe en ninguna transcripción.

✅ Output con Claude Code anclado en datos:

"El problema no era los datos, era acceder a los datos. Si pudiera haber visto mi puntaje de recuperación con solo mirar la muñeca cada mañana, creo que seguiría siendo suscriptora."
P03, [00:06:18] — cita verificada en transcripción original
✅
Cómo solucionarlo

En tu CLAUDE.md, agrega la regla: "Toda afirmación sobre un participante debe ir acompañada de una cita textual y su timestamp. Si no hay cita, escribe explícitamente que no hay evidencia directa." Con Claude Code esto se puede hacer sistemáticamente en todas las transcripciones a la vez.

2

Insights genéricos

Hallazgos que aplican a cualquier producto y no dicen nada nuevo

›
❌
Qué es

El análisis produce observaciones que cualquier stakeholder ya sabe — o que podrían aplicar a cualquier producto de cualquier industria.

⚠️
Por qué pasa

Cuando le pides a la IA "analiza estas entrevistas", sin criterio específico, el modelo busca patrones de superficie. Los patrones más fáciles de detectar son los más genéricos: "los usuarios quieren mejor UX", "la fricción reduce la retención".

❌ Insight genérico (inútil)

"Los usuarios experimentan fricción al acceder a los datos de recuperación, lo que afecta negativamente su engagement con la aplicación."

✅ Insight específico (accionable)

"Para usuarios con hijos pequeños, el proceso de 3 pasos para ver datos en la app es suficiente para romper el hábito de verificación matutina — la ventana de atención se cierra en 10-15 segundos antes de que empiecen las interrupciones del hogar."

✅
Cómo solucionarlo

Define en tu CLAUDE.md la pregunta de investigación específica. En el proyecto Whoop era: "¿Debe Whoop agregar una pantalla al wristband?". Cada insight debe responder directamente esa pregunta, citando el segmento de usuario, el contexto específico, y la implicación accionable.

3

Señal no accionable

Datos que describen el problema pero no orientan la decisión

›
❌
Qué es

El análisis reporta una cantidad (porcentaje, frecuencia) pero no dice qué hacer con esa información. El stakeholder queda igual de perdido.

Ejemplo del proyecto Whoop. La encuesta de salida tenía este dato:

❌ Señal no accionable:

"31 de 150 usuarios churneados (21%) citaron la falta de pantalla como razón principal de cancelación." — ¿Y entonces? ¿Agregas pantalla o no?

Las entrevistas en profundidad revelan lo que la encuesta no puede ver:

✅ Señal accionable (del análisis cross-entrevistas):

"El 21% que dijo 'quiero pantalla' incluye al menos 3 perfiles distintos con necesidades incompatibles: (a) padres ocupados que quieren un vistazo de recuperación en 2 segundos, (b) atletas de trail que necesitan GPS completo, (c) usuarios de fitness grupal que en realidad no pueden actuar sobre los datos independientemente del formato. Una sola pantalla no soluciona los tres casos."

"Podría haber marcado 'quiero una pantalla' en la encuesta de salida. Pero ahora tengo un Apple Watch con pantalla y no cambia nada. El problema real era otro."
P08, usuario churneado — el dato de encuesta habría sido engañoso
✅
Cómo solucionarlo

Usa el skill /workflow-decisions para que el análisis esté anclado desde el inicio a una decisión específica. Cada hallazgo debe responder: ¿apoya o refuta la decisión propuesta? ¿Para qué segmento?

4

Contradicciones no detectadas

El análisis presenta hallazgos opuestos sin notar que se contradicen

›
❌
Qué es

Un hallazgo dice "los usuarios quieren pantalla" y otro dice "los usuarios prefieren el diseño minimalista sin pantalla" — ambos verdaderos para segmentos distintos, pero presentados como verdades universales que se contradicen.

⚠️
Por qué pasa

La IA tiende a reportar lo que cada entrevista dice sin verificar si los hallazgos son consistentes entre sí. Sin una etapa de síntesis que compare explícitamente, las contradicciones pasan desapercibidas.

En el proyecto Whoop, esto fue exactamente lo que la síntesis de evidencia detectó:

"La queja de la pantalla enmascara problemas heterogéneos. P01 quería la funcionalidad de un reloj GPS. P03 quería acceso a datos sin fricción. P06 quería métricas de running en tiempo real. P07 quería interpretación de datos. P08 quería características sociales y gamificación. Los cinco podrían haber marcado 'quería una pantalla' en la encuesta de salida, pero significaban cosas completamente diferentes."
Fragmento real de evidence-synthesis.md — análisis Whoop Q4 2025
✅
Cómo solucionarlo

Usa /cross-source-triangulation y /assumption-validation después de analizar entrevistas individuales. Estos workflows fuerzan una comparación explícita entre fuentes y etiquetan cada hallazgo como "convergente" o "divergente".

5

Aplanamiento de segmentos

Agregar datos de perfiles distintos que se cancelan mutuamente

›
❌
Qué es

El análisis promedia respuestas de usuarios con necesidades opuestas. El resultado no representa a nadie — es un perfil ficticio que cancela las señales reales.

⚠️
Por qué pasa

Por defecto, la IA trata todos los participantes como homogéneos a menos que se le instruya explícitamente lo contrario. En encuestas especialmente, el "promedio" puede esconder que la distribución es bimodal.

En Whoop, mezclar P01 (ultra runner que no quiere pantalla) con P03 (madre ocupada que sí la quiere desesperadamente) produce conclusiones sin sentido. La síntesis correcta fue:

"La evidencia se divide claramente por segmento, haciendo que una respuesta agregada a '¿debemos agregar una pantalla?' sea engañosa. Para P03: sí, sin ambigüedad. Para P01: no. Para P07: no, y confirmado por experiencia propia con Apple Watch."
evidence-synthesis.md — Mapa de Evidencia por Decisión
✅
Cómo solucionarlo

Antes de analizar, define en tu CLAUDE.md los segmentos relevantes. Instrucción: "Nunca agregues hallazgos entre segmentos sin justificarlo explícitamente. Si los datos divergen por segmento, repórtalos por separado."

Detecta un error en un análisis pasado tuyo

Abre el último análisis que entregaste a un cliente. Lee los hallazgos y busca:

  • ¿Hay afirmaciones sin cita directa del participante? → Error 1: evidencia inventada

Sección 4 — Análisis profundo

4 señales que el análisis superficial pierde

Los patrones más ricos de una entrevista no están en lo que el participante dice directamente — están en cómo lo dice.

🎯

Lo que vas a aprender: Cuatro tipos de señales de alto valor que solo emergen cuando el análisis va más allá de las respuestas literales. Con fragmentos reales de transcripciones de investigación.

🏷️

Datos de ejemplo: Los fragmentos de esta sección vienen de entrevistas reales del proyecto Whoop (estudio sobre wearables fitness, usado aquí como caso de práctica). Aplica exactamente la misma lógica a tus entrevistas de clientes LVL: conversaciones del bot, sesiones de need assessment, o entrevistas de campo.

🔧

1. Workarounds — señales de fricción real

Lo que el usuario hace para compensar lo que el producto no hace

Qué buscar: Cuando un participante describe pasos adicionales, soluciones temporales, o hábitos compensatorios que inventaron para que el producto "funcione" en su vida — ese es el workaround. Es evidencia directa de un dolor que el usuario ya asumió como parte del flujo.

Por qué lo pierde el análisis superficial: El participante no lo llama "workaround" — lo describe como parte de su rutina normal. Solo cuando lo comparas con la intención del diseño se vuelve visible.

"Estaba intentando usar el celular en un brazalete para ver las métricas mientras corría. Pero con el sudor y el movimiento, era prácticamente imposible desbloquear la pantalla. Terminé memorizando mis zonas de frecuencia cardíaca antes de cada entrenamiento para no tener que mirar nada."
P06, corredor/ciclista — workaround que revela que el producto no era funcional para su deporte
"Aprendí a revisar el app la noche anterior en lugar de la mañana. Si el lunes iba a ser un día difícil según Whoop, lo tenía presente desde el domingo. No era lo ideal, pero funcionaba."
P07, consultora con viajes frecuentes — workaround temporal que eventualmente no fue suficiente
💡
Cómo detectarlo con Claude Code

Instrucción: "En cada transcripción, identifica momentos donde el participante describe haber creado un proceso adicional, hábito alternativo, o solución inventada para compensar una limitación del producto. Cita el fragmento y clasifica el workaround."

😮‍💨

2. Aceptación resignada — dolor normalizado

Cuando el usuario asumió que el problema es culpa suya, no del producto

Qué buscar: Frases como "supongo que no soy el tipo de usuario para esto", "entiendo que no es para mí", "probablemente no lo usaba bien". El participante ha internalizado el fracaso como un defecto propio, no como un fallo del diseño.

Por qué lo pierde el análisis superficial: El participante nunca lo presenta como queja — de hecho, puede incluso defender al producto. La señal está en lo que no dice: no exige mejora porque ya asumió que no va a mejorar.

"Veía el puntaje amarillo y pensaba: quizás debería tomarlo con calma hoy. Pero ya me había inscrito en la clase de las 6am. Y en Orange Theory si cancelas con menos de ocho horas de anticipación te cobran igual. Entonces iba de todas formas."

[Más tarde en la entrevista:] "Supongo que Whoop es más útil para personas que pueden controlar su propio entrenamiento. Yo no puedo — tengo un instructor."
P07 — aceptación resignada: el usuario no reconoció que el problema era un mismatch de diseño, lo internalizó como limitación propia
💡
Cómo detectarlo con Claude Code

Instrucción: "Identifica frases donde el participante asume responsabilidad por una limitación del producto. Palabras clave: 'supongo que no soy', 'probablemente yo', 'quizás el problema es que', 'entiendo que no está diseñado para'. Cada una puede indicar una fricción que el usuario ya dejó de reportar como queja."

💥

3. Picos emocionales — el momento que lo cristalizó

El episodio específico donde la frustración acumulada se hizo visible

Qué buscar: Anécdotas narradas con detalle excepcionalmente vívido. Cuando un participante recuerda una fecha, una hora, una persona, un objeto específico — esa especificidad indica que el momento tuvo carga emocional. Es la historia que se cuentan a sí mismos para justificar su decisión.

Por qué lo pierde el análisis superficial: El análisis temático agrupa estas historias bajo categorías como "fricción de acceso" y pierde la carga emocional que las hace significativas como evidencia.

"Había tenido un ride muy fuerte la noche anterior y quería ver mi recuperación y strain. Me desperté, agarré el celular, abrí el app. Y estaba sincronizando.

Mientras esperaba, mi hijo de siete años entró llorando porque su hermana le dijo que su nave Lego era estúpida. Y yo estaba ahí parado con el celular esperando un puntaje de recuperación mientras mi hijo lloraba por los Legos. Y pensé: esto es absurdo. Puse el celular, manejé la crisis Lego, y cuando todo terminó ya me había olvidado completamente de Whoop.

Ese fue el momento donde me di cuenta: esto no va a funcionar como está."
P03, Austin TX — pico emocional que cristalizó la decisión de cancelar. Este fragmento no sería visible en ninguna encuesta.

Lo que hace poderoso este fragmento no es solo la fricción de acceso — es el arco emocional completo: esfuerzo deliberado → interrupción → momento de claridad → decisión. La síntesis del proyecto lo llamó "arco culpa-alivio": el producto se convirtió en una fuente de estrés, y cancelarlo trajo alivio.

💡
Cómo detectarlo con Claude Code

Instrucción: "Identifica el momento de mayor densidad narrativa en cada entrevista — donde el participante da más detalles específicos (lugar, hora, persona, objeto). Ese es probablemente el pico emocional. Extráelo con su timestamp y explica por qué fue decisivo."

🎭

4. Contradicciones decir-hacer

Cuando lo que el usuario dice que hace no coincide con lo que describe que hace

Qué buscar: El participante afirma una intención o comportamiento en abstracto ("yo siempre reviso mis métricas") y luego describe un comportamiento concreto que lo contradice ("esa mañana no tuve tiempo, y al día siguiente tampoco"). La brecha entre el yo-ideal y el yo-real es donde viven las necesidades más honestas.

"Quería trackear todo esto correctamente. Trabajaba cinco o seis días a la semana, era el momento de hacerlo en serio."

[Más tarde:] "Honestamente nunca entendí del todo los números. ¿El strain de 20.1 era bueno? No lo sé. Mi amigo lo decía como si fuera algo impresionante, entonces yo decía 'genial.' Pero no tenía idea."
P08, Miami — declaración de intención vs. comportamiento real. La encuesta registraría "quería mejorar el tracking de salud", pero la realidad era que nunca activó esa intención.
"Me enganché inmediatamente — checaba el app todo el tiempo. Aprendí cosas, fue genial."

[Cinco minutos después:] "Después de un par de meses, internalizé lo que aprendí sobre el sueño. Ya no necesitaba seguir midiendo. Sabía lo que tenía que hacer."
P01, Courchevel — no es contradicción negativa: es la señal de que el producto completó su trabajo. Pero implica un techo natural de engagement.
💡
Cómo detectarlo con Claude Code

Instrucción: "Para cada participante, compara las afirmaciones de comportamiento declarativo ('yo siempre hago X', 'mi hábito es Y') con los episodios concretos que describe. Cuando hay discrepancia entre lo abstracto y lo concreto, márcalo como contradicción decir-hacer y extrae ambas citas."

Estructura de carpetas recomendada

⚠️
Regla de oro: Data/ es sagrada

Los archivos en Data/ son los originales. Nunca se modifican. Si necesitas limpiar datos, crea un archivo limpio en Analysis/. Esto garantiza reproducibilidad — siempre puedes volver a los datos crudos.

El CLAUDE.md: memoria permanente del proyecto

El CLAUDE.md es el archivo más importante. Claude lo lee automáticamente al inicio de cada sesión — es como el briefing que le das a un analista el primer día. Incluye: qué es el proyecto, qué pregunta estás respondiendo, y cómo quieres que trabaje.

CLAUDE.md
# Proyecto: Research Whoop — Screen Exploration Q4 2025

## Contexto del proyecto
Analizamos entrevistas cualitativas con usuarios churneados de Whoop
para responder una pregunta de decisión de producto:

**Pregunta central:** ¿Debe Whoop agregar una pantalla al wristband?

El cliente necesita evidencia que apoye o refute esta decisión,
segmentada por tipo de usuario.

## Estructura de datos
- Data/: 5 transcripciones (.txt) + 1 encuesta CSV (150 respuestas)
- Todas las fuentes son en inglés — mantener citas en inglés con
  traducción libre en paréntesis cuando sea necesario

## Reglas de análisis
1. Toda afirmación sobre un participante DEBE ir con cita textual + timestamp
2. Si no hay evidencia directa, escribir explícitamente "sin evidencia directa"
3. Nunca agregar hallazgos entre segmentos sin justificación explícita
4. Etiquetar cada hallazgo: [APOYA pantalla] / [REFUTA pantalla] / [COMPLEJO]
5. Separar siempre: qué dice el participante vs qué sugiere el comportamiento

## Segmentos definidos
- Atletas de resistencia (P01, P06): usan GPS watches, entrenan solos
- Usuarios de fitness grupal (P07, P08): clases instructoradas, no controlan intensidad
- Padres ocupados (P03): caos doméstico, ventanas de atención cortas

## Formato de outputs
- Análisis individuales: /Analysis/YYYY-MM-DD-[codigo]-[tipo].md
- Síntesis: /Analysis/YYYY-MM-DD-synthesis.md
- Cada análisis empieza con: Participante | Segmento | Disposición | Veredicto

Reglas en .claude/rules/

Las reglas en .claude/rules/ son instrucciones más específicas que se cargan según el contexto. A diferencia del CLAUDE.md, puedes tener múltiples archivos de reglas para distintos tipos de tareas.

.claude/rules/citas.md
# Reglas de citación y evidencia

## Formato obligatorio de cita
"[texto exacto de la cita]" [código participante], [timestamp o línea]

Ejemplo correcto:
"Si pudiera haber visto mi recuperación con solo mirar la muñeca,
creo que seguiría siendo suscriptora." P03, [00:06:18]

## Lo que NUNCA debes hacer
- Parafrasear sin citar ("el participante expresó que...")
- Generalizar sin base ("los usuarios sienten que...")
- Usar adverbios de frecuencia sin conteo ("frecuentemente", "siempre")

## Escala de fuerza de evidencia
- FUERTE: 3+ participantes con citas directas convergentes
- MODERADA: 2 participantes O 1 con cita muy específica y vívida
- DÉBIL: 1 participante sin cita directa o con cita ambigua
- SIN EVIDENCIA: inferencia del analista sin base en datos

Formatos óptimos de archivos

📄 Markdown (.md) — para todo análisis

  • Claude lo lee perfecto, con estructura
  • Permite headers, listas, tablas, citas
  • Legible en GitHub, Notion, Obsidian
  • Versionable con git
  • ❌ Evita Word o Google Docs — Claude no los lee bien

📊 CSV — para datos cuantitativos

  • Separador punto y coma (;) recomendado
  • Primera fila: headers claros y cortos
  • Sin celdas combinadas ni colores
  • Claude puede leer y agregar directamente
  • ❌ Evita Excel con fórmulas o macros

📝 Texto plano (.txt) — para transcripciones

  • Ideal para transcripciones crudas largas
  • Incluye timestamps en formato [HH:MM:SS]
  • Separa participantes con líneas en blanco
  • Indica speaker: "Entrevistador:" / "P01:"

🖼️ Imágenes (.png/.jpg) — para contexto visual

  • Claude puede ver y describir imágenes
  • Útil para screenshots de apps, mapas de viaje
  • Incluir siempre nombre descriptivo del archivo
  • No usar para datos tabulares — usar CSV

Sección 6 — Automatización

Skills — automatiza tu metodología

Un skill es tu metodología convertida en un comando reutilizable. Lo que tardas 20 minutos en explicar, lo ejecutas en 3 segundos.

🎯

Lo que vas a aprender: Qué es un skill, cómo usarlo, cómo mejorarlo con el ciclo pressure-test → improver, y la diferencia entre commands, skills y subagentes.

Imagina que cada proyecto de investigación empieza igual: lees las transcripciones, buscas workarounds, buscas aceptación resignada, buscas picos emocionales, buscas contradicciones decir-hacer. Eso son 4 pasadas por cada transcripción, cada vez explicando desde cero qué buscar.

Un skill es esa explicación empaquetada como un comando. Escribes /extract-signals y Claude ejecuta todas las pasadas con la misma consistencia, siempre.

Ejemplo: el skill /extract-signals

/extract-signals
Extrae las 4 señales de alto valor de una transcripción de entrevista
# Skill: extract-signals
# Propósito: analizar una transcripción y extraer señales de alto valor

Analiza el archivo de transcripción que se te indica y extrae:

## 1. Workarounds
Identifica momentos donde el participante describe haber inventado
un proceso alternativo para compensar una limitación del producto.
Formato: [timestamp] + cita + clasificación del workaround

## 2. Aceptación resignada
Busca frases donde el usuario internaliza un fallo del producto
como limitación propia. Palabras clave: "supongo que no soy",
"probablemente yo", "entiendo que no está diseñado para mí".

## 3. Pico emocional
Identifica el momento de mayor densidad narrativa — donde el
participante da más detalles específicos (hora, lugar, persona,
objeto). Ese es el episodio que cristalizó su decisión.

## 4. Contradicciones decir-hacer
Compara afirmaciones declarativas ("yo siempre hago X") con
episodios concretos. Extrae ambas citas cuando hay discrepancia.

## Formato de output
Para cada señal encontrada:
- SEÑAL: [tipo]
- CITA: "[texto exacto]" [timestamp]
- RELEVANCIA: [por qué importa para la pregunta de investigación]
- FUERZA: FUERTE / MODERADA / DÉBIL

2. Workflow-Decisions — el orquestador de 4 skills

El skill /workflow-decisions es un orquestador que encadena automáticamente 4 skills en la secuencia correcta para responder una pregunta de decisión de producto:

1️⃣ /decision-decomposition
→
2️⃣ /assumption-extraction
→
3️⃣ /cross-source-triangulation
→
4️⃣ /assumption-validation

1️⃣ Decision Decomposition

Toma la pregunta de decisión y la descompone en sub-preguntas específicas que los datos pueden responder.

2️⃣ Assumption Extraction

Extrae los supuestos implícitos en la decisión. Ej: "asume que todos los usuarios que se quejaron de pantalla tienen la misma necesidad".

3️⃣ Cross-Source Triangulation

Compara las respuestas de entrevistas con los datos de encuesta, identificando convergencias y divergencias críticas.

4️⃣ Assumption Validation

Evalúa cada supuesto contra la evidencia disponible. Resultado: supuestos confirmados, refutados, o sin evidencia suficiente.

3. El método separar-comparar-verificar

Para proyectos con múltiples fuentes de datos (entrevistas + encuesta + analytics), el método más robusto es:

  • Separar — analiza cada fuente aislada

    Primero analiza cada fuente sin mezclarlas. Esto previene que los datos cuantitativos contaminen la lectura cualitativa.

    > Analiza solo Data/encuesta-salida.csv.
  Codifica los temas sin ver las entrevistas todavía.

  • Comparar — identifica convergencias y divergencias

    Una vez que tienes análisis separados de cada fuente, compáralos explícitamente.

    > /cross-source-triangulation
  Compara Analysis/encuesta-codificada.md con
  Analysis/entrevistas-synthesis.md.
  Identifica qué coincide y qué contradice.

  • Verificar — explica las divergencias

    Las divergencias son las más interesantes. Cuando la encuesta dice una cosa y las entrevistas dicen otra, ahí está la señal más valiosa.

    La encuesta reportó que el 21% se fue por "falta de pantalla". Las entrevistas revelaron que este número incluye perfiles con necesidades incompatibles. La divergencia no invalida la encuesta — la profundiza.

4. Context rot — y cómo manejarlo

En sesiones largas de análisis, el "context rot" es el deterioro gradual de la calidad de las respuestas conforme la sesión de Claude Code se llena de información. El modelo empieza a confundir datos de distintas entrevistas, repite puntos, o pierde el hilo de la pregunta de investigación.

⚠️
Señales de context rot

• Claude empieza a mezclar citas de distintos participantes
• Las respuestas se vuelven más genéricas y menos específicas
• Claude "olvida" reglas que estaban en el CLAUDE.md
• Las respuestas se repiten o hacen referencia a puntos anteriores de forma incorrecta

🗜️ Solución 1: /compact

El comando /compact comprime el historial de la sesión actual manteniendo los puntos clave. Úsalo cada 30-40 minutos en sesiones largas.

> /compact

✂️ Solución 2: dividir sesiones

Para proyectos grandes, divide el análisis en sesiones cortas con propósito específico. Guarda el output de cada sesión en un archivo antes de cerrar.

Sesión 1: análisis individual P01-P03
Sesión 2: análisis individual P06-P08
Sesión 3: síntesis cross-participantes

🤖 Solución 3: subagentes

Cada subagente del /opportunity-finder tiene su propio contexto limpio. Para proyectos con muchos archivos, los subagentes son la mejor solución contra el context rot.

📋 Solución 4: CLAUDE.md robusto

Un CLAUDE.md bien escrito funciona como "ancla de contexto" — cuando el modelo derive, el CLAUDE.md lo recalibra en cada sesión nueva.

Lo que vas a aprender: Tres casos de uso inmediatos para LVL: análisis de conversaciones del bot de WhatsApp, encuestas con need assessment y MaxDiff, y el flujo combinado Claude Code + Claude Web para máxima eficiencia.

Caso 1: Análisis de conversaciones del bot de WhatsApp

El bot de WhatsApp genera conversaciones ricas en señales de comportamiento — pero en un formato que requiere preprocesamiento antes de que Claude Code pueda analizarlo bien. El flujo recomendado:

  • Exporta y limpia las conversaciones

    Exporta los chats en formato .txt. Estandariza el formato: cada mensaje en una línea con speaker y timestamp.

    Data/conversacion-cliente-001.txt
    # Formato recomendado para conversaciones de WhatsApp
Bot: Hola María, ¿cómo te fue con el producto esta semana?
María: Bien, aunque tuve que hacer X antes de poder usarlo
Bot: ¿Puedes contarme más sobre ese paso X?
María: Sí, es que cuando llegué a la tienda no había...

  • Define tus señales de interés en CLAUDE.md

    Especifica qué estás buscando en las conversaciones. Sin esto, el análisis será genérico.

    CLAUDE.md — sección de señales
    ## Señales de interés en conversaciones bot
- Workarounds: pasos adicionales que el cliente inventa para usar el producto
- Abandono silencioso: temas que se mencionan una vez y no se retoman
- Lenguaje de esfuerzo: "tuve que", "me tocó", "me costó", "intenté"
- Comparaciones: cuando el cliente menciona otra marca o alternativa
- Momentos de delight: exclamaciones, emojis positivos, lenguaje de sorpresa

  • Corre el análisis con subagentes paralelos

    Si tienes 20+ conversaciones, usa /opportunity-finder para analizarlas todas a la vez.

    Claude Code
    > /opportunity-finder
  Analiza todas las conversaciones en Data/whatsapp/ en paralelo.
  Pregunta: ¿Cuáles son los principales puntos de fricción
  y momentos de abandono en la experiencia del cliente?

✓ 20 subagentes completados. Output: Analysis/whatsapp-synthesis.md
💡
Señal única de WhatsApp: el lenguaje informal

Las conversaciones de WhatsApp tienen un nivel de autenticidad que las encuestas no pueden replicar. El cliente dice "eso no jalaba" en lugar de "el producto no funcionaba" — ese lenguaje coloquial es una señal más honesta. Instrucción para Claude: "Presta atención al vocabulario coloquial y a las expresiones informales — a menudo revelan la experiencia real mejor que el lenguaje formal."

Caso 2: Encuestas con Need Assessment y MaxDiff

Las encuestas de Need Assessment y MaxDiff generan datos cuantitativos — pero los errores más comunes (aplanamiento de segmentos, señal no accionable) son exactamente los que Claude Code puede prevenir.

📊 Need Assessment con Claude Code

  • Exporta los resultados como CSV (separador ;)
  • Corre /data-quality-audit antes de analizar
  • Instrucción clave: "Segmenta los resultados por [variable demográfica o comportamental relevante] antes de agregar. No me des el promedio total."
  • Usa /cross-source-triangulation para comparar con entrevistas cualitativas del mismo proyecto

🔢 MaxDiff con Claude Code

  • Los datos MaxDiff tienen formato específico — incluye un ejemplo del formato en tu CLAUDE.md
  • Instrucción clave: "No interpretes los scores MaxDiff como porcentajes de preferencia directa. Son scores relativos — compáralos entre segmentos."
  • Crea un skill específico para tu formato de exportación de MaxDiff — así cualquier encuesta futura se analiza igual

Caso 3: Claude Code + Claude Web juntos — el flujo ideal de LVL

No tienes que elegir entre uno u otro. El flujo más eficiente usa cada herramienta donde es más fuerte:

Momento del proyecto 🌐 Claude Web (claude.ai) 💻 Claude Code (terminal)
Kick-off con el clienteRedactar preguntas de investigación, refinar el brief, generar hipótesis iniciales—
Diseño de encuestaGenerar opciones para preguntas MaxDiff, revisar el cuestionario, sugerir mejoras—
Procesamiento de datos—Auditar el CSV, limpiar, analizar por segmentos, extraer señales
Análisis de entrevistasAnalizar 1-2 entrevistas rápidas para exploración inicialAnalizar todas las entrevistas en paralelo con subagentes, síntesis cross-fuentes
Síntesis de hallazgos—/workflow-decisions, /cross-source-triangulation, /assumption-validation
Entregable para clientePulir el lenguaje del reporte, adaptar el tono, revisar la narrativaGenerar el borrador inicial en markdown con citas verificadas
Preguntas de seguimientoConsultas rápidas, exploración de ideas, feedback del cliente—
✅
Regla práctica

Si puedes pegarlo en un chat → Claude Web. Si tienes archivos que procesar o necesitas consistencia entre proyectos → Claude Code. El 70% del tiempo de investigación profunda estará en Claude Code; el 30% de planificación y comunicación estará en Claude Web.

Aplica Claude Code a un proyecto LVL real esta semana

Elige un proyecto en curso o recién cerrado. Define:

  • ¿Qué fuente de datos tienes? (bot WhatsApp / encuesta / entrevistas / mezcla)
  • ¿Cuál es la pregunta de decisión del cliente? (qué tiene que decidir basándose en tu análisis)
  • ¿Qué segmentos son relevantes? (por región, perfil de cliente, canal, etc.)

Con esas tres respuestas, escribe tu CLAUDE.md del proyecto. Es el primer paso y el más importante.
